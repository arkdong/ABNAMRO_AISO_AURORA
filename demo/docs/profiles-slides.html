01 · Concept

Two axes, independent.

A profile bundles knowledge, skills, tools, and guardrails. Selection is intent-driven: a single request typically activates one profile per axis; some flows activate many.

Workflow axis

How the user is working

Stage of the editorial loop — drafting, reviewing, curating. Reusable across sectors and channels.

  • Drafter — generates content
  • Reviewer — checks against rules
  • Curator — surfaces & ages corpus
Domain expert axis

What the work is about

Sector-specific expertise expressed as role-based archetypes inferred from corpus output. New sectors plug in here without touching workflow.

  • TMT Sector Generalist — breadth across cyber / software / AI / cloud
  • TMT Media & Advertising Specialist — ad-tech, retail media, DOOH
  • TMT Cybersecurity Specialist — Cyber × AI
02 · Schema

One shape, two specialisations.

Required for every profile: id, name, description, category, activates_on, knowledge, capabilities, co_activates_with.

id:            string                              # stable identifier
name:          string                              # human-readable
description:   string                              # one-line summary
category:      "workflow" | "domain_expert"        # which axis

activates_on:                                      # selection rules
  # workflow:
  intent_codes:  [T1_DRAFT, T1_TRANSLATE, ...]
  # domain_expert:
  sector:        "Technologie, Media & Telecom"
  topic_keywords: [string, ...]

knowledge:     [string, ...]                       # sources consulted
capabilities:                                      # skills | expertise_areas
tools:           [string, ...]                     # workflow-only
guardrails:      [string, ...]                     # workflow-only
style_signature: [string, ...]                     # expert-only
outputs:         [string, ...]                     # workflow-only
co_activates_with: [profile_id, ...]
03 · Activation

Intent in, bundle out.

The intent classifier resolves both axes independently. Workflow profiles are matched on intent codes; experts on sector + keyword overlap. Bundles merge into prompt assembly.

request → intent_classifier
▼
workflow_axis → match intent_codes → 1+ workflow profiles
domain_axis → match sector + topics → 0..n experts
▼
profile_bundle = workflow ∪ experts
▼
prompt_assembly(bundle) → LLM call

If no expert matches, the bundle still ships with a workflow profile. Multiple expert matches are passed with overlap-aware merging.

04 · Workflow profiles

Three stages of the editorial loop.

drafter.yaml · T1_DRAFT, T1_TRANSLATE

Drafter

Generates new content grounded in the writing guide and approved examples. Handles NL↔EN translation. Hard rules on tone, claims, length, voice.

co-activates → curator, reviewer
reviewer.yaml · T2_COMPLIANCE

Reviewer

Checks drafts against hard rules, soft guidance, channel checklist. Never silently rewrites — surfaces structured pass/fail report and non-binding diff.

co-activates → drafter
curator.yaml · T1_SEARCH, T4_RENEWAL

Curator

Searches the corpus, surfaces prior art, scans for aging articles (>1 year). Never invents — only what already exists in the corpus.

co-activates → drafter

Workflow profiles are sector-agnostic — they survive every extension to new domains, languages, or channels.

04.1 · drafter.yaml

Drafter.

Hook → context → insight → implication. ABN AMRO Insights voice. NL/EN bilingual; British EN.

Tools
  • retrieval.fetch(query, sector, lang)
  • llm.generate(context_pack, prompt)
  • llm.translate(source, target_lang)
  • corpus.examples(sector, topic, n=3)
Guardrails
  • No unsubstantiated forward-looking claims
  • Cite internal or approved external sources
  • Max 1200 words (800 for headline pieces)
  • No "we believe" / "we think" voice
  • NL formal "u"-form · EN British spelling
Outputs
  • Markdown draft + frontmatter
  • Inline citations to retrieved sources
  • Audit trail of retrieved sources used
Knowledge
  • Writing Guide 2026 V1.1
  • Approved EN/NL Insights corpus
  • Sector-specific corpus (TMT this iteration)
  • Style examples surfaced via Curator
04.2 · reviewer.yaml

Reviewer.

Hard rules score automatically. Soft guidance scores qualitatively. Humans decide.

Tools
  • rules.check(draft, ruleset)
  • llm.score(draft, rubric)
  • readability.score(draft, target=B1)
  • diff.suggest(draft, fixed_draft)
Guardrails
  • Never silently rewrite
  • Always emit structured report
  • Cite the rule_id behind each finding
Outputs
  • { hard_rules, soft_guidance, overall: pass|fail|review }
  • Suggested edits as non-binding diff
Skills
  • Hard-rule conformance scoring
  • Tone / structure / B1 readability review
  • Source-citation verification
  • Comparative scoring vs approved examples
04.3 · curator.yaml

Curator.

The corpus is the ground truth. The Curator surfaces what exists — and what's aging.

Tools
  • corpus.search(query, sector, lang, k)
  • corpus.filter(tag, date_range, author)
  • corpus.aging_scan(threshold_days=365)
  • corpus.gap_report(topic)
Guardrails
  • Never invent articles or citations
  • Aging detected → surface original author
Outputs
  • Ranked article list (title, date, author, tags, score)
  • Coverage / gap report
  • Renewal queue (aging-flagged articles)
Knowledge
  • Insights corpus index (PageIndex tree + metadata)
  • Article metadata (date, author, sector, tags)
  • Aging policy (>1 year flagged)

07 · Scalability

The two-axis model is the extension mechanism.

Adding a sector touches only experts. Adding a channel touches only workflow guardrails. The schema, loader, and activation logic are unchanged.

ExtensionWhat changesWhat stays
New sector (Food & Agri) Add domain expert archetypes for the sector Workflow profiles unchanged
New language Drafter knowledge gains examples; Reviewer ruleset adds language rules Schema unchanged
New channel (Chatbot Anna) New Reviewer ruleset; Drafter examples; possibly a new workflow stage Domain experts unchanged
New workflow stage (Translator) New workflow profile + intent code Domain experts unchanged

TMT is the proof case — same Drafter / Reviewer / Curator routes succeed for cyber, media, and advertising sub-topics; only the activated expert changes.

08 · Machine source

Six YAMLs, one loader, one validator.

The narrative lives in docs/profiles.md. The loader reads YAML — so the system can route on data, not prose.

profiles/
├── _schema.md              # schema reference (humans)
├── loader.py               # dataclasses + load_all / load_by_id / match
├── validate.py             # `python -m profiles.validate`
├── workflow/
│   ├── drafter.yaml        ✓
│   ├── reviewer.yaml       ✓
│   └── curator.yaml        ✓
├── domain_expert/
│   ├── tmt_generalist.yaml          ✓
│   ├── tmt_media_advertising.yaml   ✓
│   └── tmt_cybersecurity.yaml       ✓
└── tests/test_profiles.py

Validator confirms every YAML parses, co_activates_with references resolve, and intent codes belong to the canonical set: T1_DRAFT, T1_TRANSLATE, T1_SEARCH, T2_COMPLIANCE, T4_RENEWAL.
